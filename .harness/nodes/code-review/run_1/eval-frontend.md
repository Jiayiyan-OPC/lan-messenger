# Frontend Code Review — run_1

## Verdict: ITERATE

## Summary
The redesign lands the handoff visual vocabulary faithfully: CSS custom properties match the spec 1:1 on hex and easing, the 3-column shell under a 26px rounded card + 36px macOS titlebar is in place, all six outcomes have concrete components wired to real Zustand stores, and the Composer implements a textbook IME guard (onCompositionStart/End + isComposing + keyCode 229). However, the Tauri window is still configured as 800×600 with the native chrome visible, which means the custom titlebar will appear below the OS titlebar and the min 1280×760 constraint from OUT-2 is not enforced — that is a 🔴 blocker for OUT-2. Secondary 🔴: the drop overlay position math divides Tauri drag-coords by DPR but the chat region rect is in CSS pixels that already ignore DPR on most desktop configs, so the hit-test is wrong on HiDPI screens. The remaining issues are 🟡/🔵 polish: unread count is hardcoded to 0, ConvoRow time overflow, the file-bubble cancel button is not wired to the store, orphan `typing-bounce` keyframe, and the legacy `useAppStore.activeTab` is kept only for a store test that no longer reflects real state.

## Findings

### 🔴 Critical

- [OUT-2] `frontend/src-tauri/tauri.conf.json:13-21` — Window declares `width: 800, height: 600` with no `minWidth`/`minHeight`/`decorations: false`. OUT-2 requires min 1280×760 and the custom 36px titlebar on top of a frameless window. Right now the inner card will render with the native OS chrome still visible and the inner 26px radius will collide with the OS frame. Fix: set `"width": 1280, "height": 800, "minWidth": 1280, "minHeight": 760, "decorations": false, "titleBarStyle": "Overlay", "hiddenTitle": true, "dragDropEnabled": true` (and on macOS enable `transparent` if you want the shadow to show through).
- [OUT-5] `frontend/src/components/Chat/index.tsx:58-63` — DPR conversion on Tauri drop-coords is incorrect. Tauri v2's `onDragDropEvent.position` is reported in *physical* pixels on Windows/Linux but in *logical* CSS points on macOS (and is already pre-scaled). Dividing unconditionally by `window.devicePixelRatio` under-reports x/y on macOS Retina, so on 2x displays the hit-test will reject drops inside the chat region and on Windows the number is right. Fix: either drop the DPR scaling entirely and rely on the window position already being CSS-space in Tauri v2 (confirmed by Tauri `onDragDropEvent` docs), or branch on platform. Simplest correct form: `const pos = await wv.outerPosition()` is *not* needed; use `const factor = await getCurrentWindow().scaleFactor(); const x = payload.position.x / factor; const y = payload.position.y / factor`. Using `window.devicePixelRatio` as the divisor is also unreliable across monitors.

### 🟡 Warning

- [OUT-3] `frontend/src/components/Sidebar/index.tsx:43-45` + `ConvoList.tsx:72,84` — Unread badge is hardcoded to `0` on every ConvoRow and the "会话" tab badge is `chatBadge={0}`. Handoff shows unread capsules (accent `#4E9AB8` tint) driven by unread state; acceptance §OUT-3 explicitly requires it. Fix: derive per-contact unread in the messages store (`messagesByContact[id].filter(m => m.sender_id === id && m.status !== 'read').length` as a starting point, or track a `readAt` timestamp per convo that persists to localStorage and resets on `setActiveConvo`). Sum those for `chatBadge`.
- [OUT-4] `frontend/src/components/Chat/FileBubble.tsx:153-158` — "取消" button has no onClick. Cancelling a transfer is in the spec ("取消按钮主色文字") and the store has no `cancelTransfer` action to back it. Fix: add `cancelTransfer(transferId)` to `stores/transfers.ts` that calls a new `cancel_file_transfer` Tauri command (or at minimum removes the transfer locally + emits a user-visible toast), then wire `onClick={() => cancelTransfer(transfer.id)}` here.
- [OUT-5] `frontend/src/components/Chat/FileBubble.tsx:117-126` — The download button on completed transfers is decorative; there's no onClick / href. OUT-5 says "on completion the download button opens the finished file". Fix: surface `transfer.local_path` through the store and either `open(path)` via the Tauri `opener` plugin or `revealItemInDir` from `@tauri-apps/plugin-opener`.
- [OUT-4] `frontend/src/stores/messages.ts:38-53` — `message-sent` handler matches by `msg.id`, but `sendMessage` (L67-80) appends the already-returned `msg` object immediately. If the backend emits `message-sent` with the same id carrying updated status that's fine — but the handler also silently drops the event when id is not found. With optimistic sends + out-of-order delivery this will drop delivered transitions. Fix: if not found, append (or at minimum log). Also consider setting `status: 'sending'` optimistically, then letting `message-sent` upgrade it to `sent`/`delivered`.
- [OUT-6] `frontend/src/components/Detail/FileList.tsx:15-20` — `belongsToPeer` falls back to "show every transfer without a peer_id". Since backend doesn't populate `peer_id` at all right now (noted in build handshake followups), the detail panel's Files tab will show **every** file exchange across every peer. At a minimum gate this behind a TODO comment pointing at the Rust-side fix and scope the panel to "最近的传输" to avoid misleading the user.
- [OUT-3] `frontend/src/components/Sidebar/ConvoRow.tsx:40-45` — On long peer names the time is flex-pushed off-screen (the name gets `flex-1` and `truncate`, time is just `shrink-0`). This works, but with `min-w-0 flex-1` on the name div and `items-baseline`, when `pinned` is true the Pin icon sits between name and time with no gap guarantee. Low-risk but visually the time can touch the pin. Fix: add `ml-auto` on the time span or wrap name+pin in their own min-w-0 container.
- [OUT-4] `frontend/src/components/Chat/MessageList.tsx:80` — `fresh` detection uses `prevLenRef.current < messages.length`. On conversation switch from a 100-msg history to a 5-msg history, prevLen=100, newLen=5 → fresh=false, which is correct; but the **reverse** (5→100, e.g. peer-switch to a long history) animates the last bubble once on mount, not a big deal. The bigger concern: when `messages` identity changes but length is equal (message edit), nothing re-animates. Acceptable, but the `prevLenRef.current` write happens inside `useLayoutEffect` while `fresh` is read during render — StrictMode's double-invoke of render will compute fresh twice with the same prevLen value; first render's `fresh=true` is fine, but in dev you'll see the pop twice on initial mount. Fix: track "seen message ids" in a Set ref and mark fresh when `!seen.has(id)`.
- [OUT-4] `frontend/src/components/Chat/index.tsx:120-134` — HTML5 `onDragLeave` clears `dragOver` to false whenever `currentTarget === target`, but moving the cursor across child elements inside the chat region can fire leave events. This is the classic flicker bug. Since the Tauri listener is the production path, the HTML5 path only matters in dev preview — noted, but worth a comment or a proper counter (enter count > 0).
- [OUT-2] `frontend/src/App.tsx:33-34` — `minWidth: 1232, minHeight: 712` — these are 1280−48 and 760−48 (subtracting the 24px outer padding on both sides). That's the inner card size; a11y and layout-wise fine, but the *outer* element has no width clamp, so on a 1200×700 window the chat card will still try to honor its 1232×712 minimum and trigger horizontal scroll of `#root` (which does have `min-width: 1280px` in index.css at L70). Combined with the tauri.conf.json 800×600 window, the `#root` min-width will cause an initial horizontal scrollbar at startup. Fix: enforce the 1280×760 minimum in the Tauri config (see first 🔴) — then this is moot.
- [OUT-3] `frontend/src/components/Sidebar/index.tsx:22` — `deviceCount = contacts.filter((c) => c.online).length` computes across all contacts including offline ones from DB that might not be in the LAN. Handoff spec for the device tab badge reads "发现 N 台" which matches online peers, so the semantics are right, but on cold start before discovery has emitted `peer-found` the badge will show `0` while the contacts.getAll() call still populated DB rows with `online: false`. Acceptable, but confirm this is desired (vs showing total contacts count).
- [OUT-7] `frontend/src/hooks/useDeviceInfo.ts:27` — `console.warn` left in production for `get_device_info` failures. Quality Constraints requires "No console errors during normal usage"; technically this only fires on failure, but in Tauri dev mode the first render can race the backend init. Fix: push a toast via `useUiStore.pushToast` instead of console, or gate on `import.meta.env.DEV`.

### 🔵 Suggestion

- `frontend/src/stores/app.ts` + `frontend/src/types/index.ts:82` — `useAppStore.activeTab` + `ActiveTab` are kept only to preserve `stores/__tests__/app.test.ts`. The UI no longer uses `activeTab`; only `deviceId`/`deviceName` are consumed (ChatView, useDeviceInfo). Replace the app store with a tiny `useDeviceStore` and delete the legacy test rather than carrying dead code.
- `frontend/src/index.css:131-134` — `@keyframes typing-bounce` is orphaned; TypingIndicator was removed (confirmed zero matches in `src/`). Delete the keyframe block to keep the stylesheet lean.
- `frontend/src/components/Chat/index.tsx:110-116` — The Attach button is a "请拖拽" toast. The handoff spec lists "附件按钮 — 模拟上传一个 PDF" as prototype-only; production should either hide it or wire `@tauri-apps/plugin-dialog` `open()` to call `sendFile(peer, path)`. Leaving a no-op toast there will feel broken.
- `frontend/src/components/Chat/MessageList.tsx:45-49` — Snap-to-bottom on `peer.id` change uses `useLayoutEffect`, but on first mount both `peer.id` and the `messages` effect fire, so scroll is set to scrollHeight twice. Low-cost but merge into one effect keyed on `[peer.id, messages.length]` with a "switching peer" flag.
- `frontend/src/components/Chat/FileBubble.tsx:107-115` — `bytesTransferred / file_size` is displayed with `formatSize(bytesTransferred)`; for a just-started transfer `file_size` is 0 (store appends with `file_size: 0` at L88) and the denominator reads "0 B" for a moment. Consider hiding the slash until file_size>0 lands via the first progress event.
- `frontend/src/components/Sidebar/DeviceList.tsx:29` — `contacts.filter(...)` inside the component body runs every render. Wrap in `useMemo` on `[contacts, searchQuery]` (same pattern you already use in `ConvoList.tsx:36`).
- `frontend/src/components/Chat/index.tsx:18` — `contacts.find()` inside `useMemo([contacts, activeConvoId])` is fine, but subscribing to whole `contacts` array causes re-render on every `peer-found`. Prefer `useContactsStore((s) => s.contacts.find((c) => c.id === activeConvoId) ?? null)` with a shallow equality function, or store `peersById` as a second map in the contacts store.
- `frontend/src/components/Detail/index.tsx:18-20` — `fileCount` selector filters in the subscribe body, so every transfers update triggers the panel re-render (shallow equality would consider the returned number stable, but you're reading a new filtered result). Use `useTransfersStore((s) => s.transfers.filter(...).length, shallow)` or derive a memoized selector outside the component.
- `frontend/src/stores/ui.ts:108-117` — Toast auto-dismiss uses `setTimeout` but never clears on manual dismiss, so clicking × on a 3.8s toast still runs a no-op timer 3s later (checked with `cur.some`). Minor — just noting.
- `frontend/src/components/Overlays/DropOverlay.tsx:16` — The 3px dashed border + 24px radius lives on an inner element inset 16px from the chat region. Works, but the handoff's DropOverlay covers the full chat area with no inset; `p-4` here creates a visible gap. Consider `p-2` or `p-0` for closer fidelity.
